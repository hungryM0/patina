## 2. 背景与问题定义

### 2.1 当前已暴露的问题

- [x] 正式版 `v0.3.2` 的持续参与识别仍然过度依赖“单次采样瞬时命中”
- [x] `GSMTC` 媒体信号存在间歇性缺失，不能作为唯一稳定来源
- [x] 音频会话兜底已接入，但在多进程媒体应用中仍可能与当前前台窗口短暂失配
- [x] 当前链路对“信号抖动、窗口短暂切换、辅助进程抢前台”的容错不足
- [x] 当前排障可观测性不足，现场很难直接判断是“没信号”还是“匹配失败”
- [x] 该问题不是抖音专属问题，而是会推及浏览器、Electron 客户端、会议软件、多进程播放器等一类应用

### 2.2 本长期线要解决的核心问题

- [x] 将“持续参与”从瞬时条件判断，升级为可解释、可观测、抗抖的长期机制
- [x] 将“当前窗口是谁”和“媒体信号属于谁”的匹配，从单进程名比较升级为稳定应用身份判断
- [x] 将 `GSMTC`、音频会话、前台窗口等多路信号收敛为统一的领域判定
- [x] 让系统对一类媒体应用稳定工作，而不是不断为单个应用追加特判

### 2.3 本长期线不解决的问题

- [x] 不承诺“所有低交互内容都能被智能识别”
- [x] 不引入摄像头、人脸、注意力检测等高侵入方案
- [x] 不把持续参与能力扩展为云端识别、远程服务或账号体系
- [x] 不把规则散落到前端页面、文案 helper 或页面级特判中
- [x] 不以“给抖音补一个名字规则”代替通用机制建设

## 3. 长期目标

长期完成态应满足以下结果：

- [x] 持续参与识别具备稳定状态机，而不是每轮 loop 独立瞬时判定
- [x] 多路信号采集、融合、解释、诊断职责明确，owner 清晰
- [x] 多进程媒体应用在合理范围内可以稳定归并到统一应用身份
- [x] 用户和开发者都能看到“为什么命中”或“为什么没命中”
- [x] 对浏览器视频、Electron 媒体客户端、会议软件、本地播放器具有统一处理框架
- [x] 新应用接入优先通过“能力画像和身份归一”完成，而不是追加散落白名单
- [x] 新机制在正确性、可解释性、发布验证上达到可进入正式版的门槛

## 4. 产品口径与最终语义

### 4.1 最终用户语义

- [x] `连续关注窗口` 负责普通交互场景中的短中断连续性判断
- [x] `持续参与时长` 只负责低交互但真实参与的媒体场景延长
- [x] “允许持续参与”不等于“当前已经处于持续参与状态”
- [x] 只有命中持续参与判定状态机，才允许按持续参与时长延长
- [x] 切到其他应用时，中间时间永远归实际切过去的应用
- [x] 是否仍属于同一轮连续关注，取决于是否在 `连续关注窗口` 内回到原应用

### 4.2 最终工程语义

- [x] `platform/windows/*` 只负责原始信号采集，不承接产品语义
- [x] `domain/tracking.rs` 负责持续参与的领域判定、身份归一、信号解释
- [x] `engine/tracking/runtime.rs` 负责时序、状态机推进、过渡与封口
- [x] `data/*` 负责状态持久化与读模型所需字段，不在数据库层偷放产品判断
- [x] 前端只消费后端已定性的 tracking snapshot 和 read model，不再本地复刻核心判定

## 5. 架构边界与 owner

### 5.1 真正 owner

- [x] `src-tauri/src/platform/windows/media.rs`
- [x] `src-tauri/src/platform/windows/audio.rs`
- [x] `src-tauri/src/platform/windows/foreground.rs`
- [x] `src-tauri/src/domain/tracking.rs`
- [x] `src-tauri/src/engine/tracking/runtime.rs`
- [x] `src-tauri/src/engine/tracking/continuity.rs`
- [x] `src-tauri/src/data/*` 中与 session/read model 直接相关的真实 owner 文件

### 5.2 明确不应成为 owner 的位置

- [x] `src/app/*` 不承接持续参与核心规则
- [x] `src/shared/*` 只保留稳定共享语义与只读文案，不成为临时规则堆放层
- [x] `src/platform/runtime/*` 不新增页面级“二次判定”
- [x] `src-tauri/src/commands/*` 不回长业务逻辑
- [x] `src-tauri/src/lib.rs` 不回长 tracking 规则编排

### 5.3 需要先停下来重审的信号

- [x] 为单一应用追加名称特判才能通过
- [x] 需要把更多规则复制到前端才解释得通
- [x] 需要把平台信号直接透传成最终业务结论
- [x] 需要引入新的共享层或兼容壳才能继续推进
- [x] 一个阶段尚未可解释，就急着叠加下一个信号来源

## 6. 分阶段执行清单

## 6.1 阶段 A：建立诊断与可观测基线

目标：

- [x] 让“为什么没进入持续参与”能够被现场定位
- [x] 在不改变最终产品语义的前提下，先获得稳定排障能力

执行项：

- [x] 为 tracking loop 增加结构化诊断快照
- [x] 明确记录当前前台窗口身份：`exe / pid / title / canonical identity`
- [x] 明确记录当前 `GSMTC` 信号：`source_app_id / playback_status / media kind`
- [x] 明确记录当前音频会话信号：`process / pid / active state`
- [x] 明确记录最终判定结果：`inactive / candidate / active / grace / expired`
- [x] 明确记录未命中的直接原因：`no_signal / inactive_signal / identity_mismatch / expired / afk_boundary`
- [x] 将诊断输出限制在可控开关下，避免常态噪音
- [x] 决定诊断数据是仅日志可见，还是同时暴露给调试面板

阶段验收：

- [x] 同一现场可以明确区分“没有媒体信号”和“有信号但匹配失败”
- [x] 复现场景时可以还原一次状态变化链，而不只是看到最后结果
- [x] 抖音、浏览器视频、会议软件至少各能得到一份完整诊断样本

## 6.2 阶段 B：建立统一应用身份与家族归一

目标：

- [x] 把“窗口是谁”和“信号属于谁”的匹配从单进程比较升级为稳定 identity 判断

执行项：

- [x] 定义领域层 `AppIdentity` 或等价稳定结构
- [x] 定义“前台窗口 identity”和“媒体信号 identity”的统一表示
- [x] 定义“主程序 identity”与“辅助进程 identity”的可控归并规则
- [x] 为多进程媒体应用建立“应用家族”归一能力
- [x] 为浏览器场景定义前台浏览器 identity 与媒体播放 identity 的匹配边界
- [x] 约束 identity 规则来源，避免页面级散落字符串匹配
- [x] 为 identity 归一失败提供明确诊断原因

阶段验收：

- [x] 抖音类多进程窗口可稳定归到同一身份
- [x] 至少一类 Electron 应用和一类会议应用可复用同一 identity 机制
- [x] 新增媒体应用接入时，优先改 identity 规则而不是追加业务特判

## 6.3 阶段 C：将持续参与升级为状态机

目标：

- [x] 将当前“每轮独立判断”升级为可维护的时序状态机

建议状态：

- [x] `inactive`
- [x] `candidate`
- [x] `active`
- [x] `grace`
- [x] `expired`

执行项：

- [x] 明确每个状态的进入条件
- [x] 明确每个状态的保持条件
- [x] 明确每个状态的退出条件
- [x] 明确普通模式与持续参与模式切换时的封口语义
- [x] 明确状态机与 `连续关注窗口`、`持续参与时长`、`AFK` 的关系
- [x] 明确切应用、短暂切出、信号瞬断时的状态转移
- [x] 为状态机输出稳定的 sealed reason 与 tracking reason

阶段验收：

- [x] 任一状态转移都能被日志和测试解释
- [x] 状态机规则不再依赖“看某一拍结果猜语义”
- [x] 普通模式与持续参与模式的边界在测试中可直接断言

## 6.4 阶段 D：做多信号融合与抗抖策略

目标：

- [x] 让持续参与识别具备对瞬时缺失和短暂抖动的容错

执行项：

- [x] 定义 `GSMTC` 信号优先级
- [x] 定义音频会话作为兜底信号的使用条件
- [x] 定义多信号冲突时的裁决顺序
- [x] 引入短时 `grace window`，避免 1 到 2 个采样周期瞬断直接退场
- [x] 区分“信号短暂缺失”和“明显已退出持续参与”
- [x] 避免 grace 被误用为无上限补时
- [x] 为不同信号来源保留来源标签，支持后续分析命中率

阶段验收：

- [x] `GSMTC` 短暂失效但音频仍在时，不会立即错误退出
- [x] `GSMTC` 与音频都短暂缺失但迅速恢复时，系统可平滑保持状态
- [x] 信号持续缺失时，系统会在明确边界退出，而不是悬挂旧状态

## 6.5 阶段 E：读模型、展示与诊断面板对齐

目标：

- [x] 让后端最终判定可以被用户和开发者理解，而不是停留在内部实现

执行项：

- [x] 明确哪些状态进入用户可见的“当前活跃状态”
- [x] 为设置页或调试视图提供只读持续参与状态展示
- [x] 明确历史页、专注时间流、应用分布对持续参与状态的消费方式
- [x] 保证展示层不重复实现状态机逻辑
- [x] 明确哪些诊断信息仅开发调试可见，哪些可以安全暴露给用户
- [x] 为后续用户反馈准备最小可解释文案

阶段验收：

- [x] 用户可以知道“当前是否处于持续参与”
- [x] 开发调试视图可以直接看到命中来源和失败原因
- [x] 前后端没有重复定义持续参与语义

## 6.6 阶段 F：测试、回归与发布收口

目标：

- [x] 将新机制收口到可进入正式版的验证水平

执行项：

- [x] 为状态机补齐 Rust 单元测试
- [x] 为关键时序补齐 replay 测试或等价回归测试
- [x] 为 identity 归一补齐多进程应用测试样本
- [x] 为 `GSMTC` 缺失、音频兜底、双信号瞬断恢复等场景补齐测试
- [x] 为普通模式和持续参与模式的封口规则补齐边界测试
- [x] 为切应用、窗口内返回、超窗返回补齐 continuity 断言
- [x] 为读模型和 UI 消费补齐前端测试
- [x] 准备发布说明，明确用户语义变化与已知边界

阶段验收：

- [x] 长期方案进入正式版前，满足最低验证门槛
- [x] `npm test`
- [x] `npm run test:replay`
- [x] `npm run build`
- [x] `cargo check`
- [x] 架构影响较大的阶段，在进入发布线前补充必要人工回放验证

## 7. 横切约束

### 7.1 产品约束

- [x] 不把持续参与能力扩展成新的复杂设置矩阵
- [x] 不增加每应用一套独立时间阈值配置
- [x] 不为了少量样例提升命中率而牺牲整体可解释性

### 7.2 架构约束

- [x] 不在前端页面复制 Rust 状态机
- [x] 不让 `commands/*` 或 `lib.rs` 重新变厚
- [x] 不让 `shared/*` 退化成规则中转层
- [x] 先判断 owner，再决定文件落点

### 7.3 发布约束

- [x] 长期方案不要求一次性全量落地
- [x] 每个阶段都应能独立收口，不把半成品直接压进稳定版
- [x] 进入正式发布前，必须说明“解决了什么”和“仍未解决什么”

## 8. 推荐推进顺序

建议顺序如下：

- [x] 先做阶段 A：诊断与可观测
- [x] 再做阶段 C：状态机基础骨架
- [x] 再做阶段 D：多信号融合与抗抖
- [x] 再做阶段 B：应用身份与家族归一
- [x] 然后做阶段 E：读模型与诊断展示
- [x] 最后做阶段 F：测试、发布、文档收口

说明：

- [x] 阶段 A 应优先，因为没有可观测性，后续优化只能靠猜
- [x] 阶段 C 与阶段 D 应尽早进入主链，因为 `v0.3.2` 的核心问题是“瞬时判定 + 不抗抖”
- [x] 阶段 B 虽然重要，但应在已有诊断和状态机骨架后推进，避免过早做大 identity 规则却无法验证收益
